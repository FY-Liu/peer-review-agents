### One-Sentence Summary
DTR is an agentic framework that treats complex, long-horizon analytical reasoning over unstructured tables as a closed-loop decision-making process, using a hierarchical meta graph for table representation, expectation-aware path selection for planning, and a siamese structured memory for continual experience-driven refinement.

### Claimed Contributions
- **Task Formalization**: Defines "Deep Tabular Research (DTR)" — long-horizon, multi-hop analytical reasoning over unstructured, non-canonical tables with hierarchical/bidirectional headers, as distinct from conventional TableQA.
- **Closed-Loop Agentic Framework**: Decouples macro-level strategic planning (operation path selection) from micro-level execution (individual code operations), treating tabular reasoning as iterative decision-making informed by accumulated feedback.
- **Experience-Driven Optimization**: An expectation-aware selection policy (UCB-style scoring over operation paths) and a siamese structured memory — parameterized execution feedback + abstracted textual experience — enabling continual refinement.
- **Empirical Validation**: DTR-Bench (500 analytical queries from RealHitBench tables) + RealHitBench, showing gains over table-specific LLMs, general LLMs, and prior agentic frameworks.

### Lens 1 — Scientific Peer Reviewer (Lead Reviewer's integrated reading)

**Soundness: 2/4** — The theoretical contribution is undermined by a mathematical error in the "Theoretical Boundedness" theorem: the claimed bound $\mathcal{E}(\pi) \le R_{\max} + \alpha \sqrt{\log \sum_{\pi'} N(\pi')}$ grows with execution experience (e.g., ~27.3 with 1000 total executions, ~31.4 with 10000), which is not a bound at all. The correct bound is $R_{\max} + \alpha$. Additionally, the structural prior $P(\pi)$ and reward mapping $\phi$ are referenced throughout key equations but never operationalized, making core algorithms unverifiable. Experimentally, zero variance or significance tests are reported, and Win Rate values >1.0 are mathematically incoherent with the paper's own definition.

**Presentation: 2/4** — Writing within sections is clear and well-organized, with good use of figures and algorithmic boxes. However, the absence of a dedicated Limitations section and the placement of related work in the appendix violate ICML presentation norms. The Win Rate metric definition is internally inconsistent (described as a proportion but taking values up to 42.74).

**Originality: 3/4** — The task formalization of "Deep Tabular Research" is a genuine contribution — naming this problem and positioning it as distinct from TableQA is valuable. The specific composition of meta-graph + expectation-aware path selection + siamese memory for unstructured tables is a moderate-level contribution built on established prior art (EXPEL, Chain-of-Table, UCB bandits). The ablation honestly shows that the experience-driven components (expectation + abstraction) contribute only ~1.3pp out of a 4.0pp total gain, suggesting the framing overstates novelty of the most distinctive mechanisms.

**Significance: 3/4** — The task formalization has real value for the tabular-reasoning subfield. The empirical finding that separating macro planning from micro execution (4.78 LLM calls vs. 8.8 for CodeLoop) improves both quality and efficiency is a useful empirical contribution. However, the gains are modest (~4pp on DTR-Bench, variable on RealHitBench), the method is heavily backbone-dependent (27-point EM gap between DeepSeek-V3 and Qwen3-4B), and the scope is narrow to tabular-reasoning agents.

**Recommendation: Weak Reject** — The paper addresses a real problem and the task formalization is valuable, but the mathematical error in the theoretical boundedness claim, the undefined P(π) and φ components, and the statistically incoherent Win Rate metric are load-bearing issues that collectively undermine the paper's credibility. The single change that would most strengthen the paper: fix the theoretical bound and define P(π) operationally.

### Lens 2 — Archaeologist (Prior Work)

**Older anchor**: Zhao et al. — "EXPEL: LLM Agents Are Experiential Learners" (AAAI 2024). Relationship: DTR's siamese memory mechanism is a direct specialization of EXPEL's core insight (LLM agents learning from execution experiences without parametric updates) to the tabular reasoning domain. Delta: EXPEL is general tool-use; DTR is domain-specific with a hierarchical meta graph, structured operation space, and tabular-oriented operation bank.

**Newer/concurrent neighbor**: Wang et al. — "Chain-of-Table: Evolving Tables in the Reasoning Chain for Table Understanding" (ICLR 2024). Relationship: Both use iterative table transformation as reasoning steps; both aim to reduce error propagation. Delta: Chain-of-Table operates on well-structured WikiTQ/FeTaQA/TabFact tables; DTR targets unstructured tables with hierarchical/bidirectional headers. Chain-of-Table uses in-context operation selection; DTR uses learned expectation-driven path selection with memory. Chain-of-Table does not have an experience/memory mechanism.

**"First to X" assessment**: "First to use experience-driven path selection for tabular reasoning" is likely valid — no prior work specifically applies UCB-style expectation-aware path selection over operation sequences for table reasoning. The siamese memory combination is novel framing but individual components exist separately in prior work.

### Lens 3 — Academic Researcher (Forward Leverage)

One concrete non-trivial follow-up project this paper enables: **Training a learned path-outcome predictor to replace the hand-crafted UCB exploration bonus.** The predictor would take path structure features + table meta graph features + query embedding → predict $R(\pi)$ before execution, replacing the current $\alpha \cdot P(\pi) \sqrt{\log \sum N(\pi')/(1+N(\pi))}$ term. DTR's paper provides the path-level reward formalization, the execution feedback instrumentation, and the scoring equation insertion point — all necessary for generating training data. This follow-up is non-trivial because it requires designing the feature space, collecting training data from diverse tables, and validating that the learned predictor outperforms the UCB prior on out-of-distribution table structures. It is materially dependent on this paper's specific path-abstraction and feedback infrastructure.

### Lens 4 — Industry Practitioner (Deployment)

**Deployment pitch**: A financial analyst's "spreadsheet intelligence" dashboard for complex unstructured Excel workbooks (merged cells, hierarchical headers, multi-sheet workbooks) — natural-language multi-hop analytical queries (e.g., "What is the quarter-over-quarter revenue growth by product category for the APAC region?") translated into executable operation paths via DTR. Positive impact: reduces manual Excel manipulation time for analysts handling complex, non-canonical spreadsheet layouts. Negative impact: 62-second average runtime per query makes interactive use impractical; the cold-start memory problem (500 queries to converge) means individual or small-batch queries receive no benefit.

**Concrete deployment blockers**:
1. **Latency**: 62-second average runtime (Table 1, DTR-Bench) is incompatible with interactive deployment at human timescales. Production spreadsheet tools require sub-5-second response.
2. **Cold-start memory**: The siamese memory requires 500+ repeated queries on the same table to converge on good operation paths. Production use involves one-off analytical queries over novel tables — the memory never converges.
3. **Backbone dependency**: DTR requires DeepSeek-V3-class performance; smaller models show 27-point EM degradation. Edge deployment is not viable.

### Lens 5 — Hacker (Reimplementability)

**Well-specified pieces**:
- The three-stage pipeline (meta graph → query-decomposed operation mapping → expectation-aware path selection → siamese memory → majority vote) is clearly described in methods.tex
- The operation bank O = {CLEAN, FILTER, GROUP, AGG, JOIN, SORT, ...} is explicitly listed
- The [THINK]/[CODE] prompting protocol is described with examples
- The expectation update rule $\hat{R}(\pi) \leftarrow (1-\eta)\hat{R}(\pi) + \eta \cdot R(\pi)$ is clear

**Missing or ambiguous pieces**:
1. **Reward mapping φ undefined**: $r(\pi) = \phi(\mathbf{f}(\pi))$ where $\mathbf{f}(\pi) = (f_{\text{exec}}, f_{\text{time}}, f_{\text{type}})$ — φ is never specified. Without it, the learning signal is absent and the algorithm cannot be reimplemented.
2. **P(π) prior un-instantiated**: The structural plausibility prior appears in the scoring equation and theoretical bound but is never computed or initialized. Reimplementers must guess.
3. **LLM decision-making protocol absent**: The paper relies on an LLM agent for operation mapping and [THINK]/[CODE] flagging, but provides no system prompt, input format, or model specification per component.

**Estimated reimplementation difficulty: Hard** — The algorithmic skeleton is clear, but three load-bearing components (φ, P(π), LLM protocol) are unspecified, requiring significant design decisions from the reimplementer.

### Integrated Verdict

The paper's most valuable contribution is the **task formalization** of Deep Tabular Research (DTR) — naming the problem of long-horizon, multi-hop analytical reasoning over unstructured tables as distinct from conventional TableQA. This is a genuine service to the subfield. The empirical finding that separating macro-level operation-path planning from micro-level code execution (4.78 vs. 8.8 average LLM calls) improves both quality and efficiency is also useful.

However, the paper has three load-bearing issues that pull toward rejection: (1) the "Theoretical Boundedness" theorem is mathematically incorrect — the claimed bound grows with execution experience rather than being a constant, which undermines the paper's theoretical framing; (2) the experimental evaluation uses a Win Rate metric that takes values >1.0 despite being defined as a proportion, making the main results mathematically incoherent; and (3) critical components — the structural prior P(π) and reward mapping φ — are referenced throughout key equations but never operationalized, preventing independent verification.

**Trade-off**: Novelty and significance pull toward accept (genuine task formalization, real empirical gains); technical soundness and experimental rigor pull toward reject (mathematical error, undefined components, statistically incoherent metrics). The paper lands at Weak Reject because the soundness concerns are load-bearing for the paper's most distinctive claims.

### Specific, Actionable Feedback

1. **[methods.tex, lines 55-75] Fix the theoretical boundedness theorem**: Replace the incorrect bound $\mathcal{E}(\pi) \le R_{\max} + \alpha \sqrt{\log \sum_{\pi'} N(\pi')}$ with the correct constant bound $\mathcal{E}(\pi) \le R_{\max} + \alpha$. The current bound grows with execution budget and is not a bound in any meaningful sense.

2. **[methods.tex, line 112] Define the reward mapping φ**: Specify exactly how $f_{\text{exec}}$, $f_{\text{time}}$, and $f_{\text{type}}$ are combined into the scalar reward $r(\pi)$. Without this, the expectation update equation is unverifiable.

3. **[methods.tex, line 41] Operationalize P(π)**: State explicitly how the structural plausibility prior is computed or initialized. The theoretical boundedness proof depends on P(π) ∈ [0,1] as an explicit assumption.

4. **[exp.tex, Table 1] Fix or remove Win Rate values >1.0**: Either redefine the metric to match what is actually computed (and justify the redefinition), or remove these values and replace with standard accuracy metrics with error bars.

5. **[exp.tex, Table 2] Investigate and explain ECR = 100.00**: A perfect execution correctness rate on Chart/Report Generation is statistically implausible for a code-generation system on unstructured tables. Report N and verify the computation is correct.

6. **[exp.tex, all tables] Report variance across ≥3 seeds**: Add standard deviation or standard error bars to all main results and ablations. Perform significance tests (paired t-test or Wilcoxon) for key comparisons.

7. **[DTR.tex] Add a dedicated Limitations section**: Disclose that (a) DTR requires frontier-level LLM backbones (27-point EM gap between DeepSeek-V3 and Qwen3-4B), (b) the operation bank may not cover all analytical intents, and (c) DTR-Bench was generated using a variant of the evaluation backbone (DeepSeek-3.2 for generation, DeepSeek-V3 for evaluation).

### Questions for Authors

1. **Theoretical boundedness**: The paper claims $\mathcal{E}(\pi) \le R_{\max} + \alpha \sqrt{\log \sum_{\pi'} N(\pi')}$ but this grows with total executions. Is the correct bound $R_{\max} + \alpha$? If so, please correct the theorem and remove the incorrect derivation.

2. **P(π) operationalization**: How is the structural plausibility prior P(π) computed or initialized in practice? Without this, readers cannot verify the expectation score or reimplement the method.

3. **DTR-Bench contamination**: DTR-Bench queries were generated using DeepSeek-3.2 and the evaluation uses DeepSeek-V3 as the backbone. Can you confirm that DTR-Bench test instances are disjoint from RealHitBench test instances used in training/pretraining of DeepSeek-V3? What data contamination checks were performed?
